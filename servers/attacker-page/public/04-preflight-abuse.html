Exercise 03 — Preflight Abuse & Private Network Access

Bugs #7, #8, #9 — advanced CORS mechanics and the 2026 attack surface
This exercise covers: How preflight caching creates a window of opportunity after a CORS fix, how listing unsafe HTTP methods in CORS headers enables cross-origin CSRF, and how Chrome's Private Network Access enforcement (new in 2024-2026) can be bypassed if the server responds carelessly to PNA preflights.

BUG #7 Preflight Over-Cached (Max-Age: 86400)

The OPTIONS response includes Access-Control-Max-Age: 86400 (24 hours). If a security team patches a CORS misconfiguration, any browser that cached the old permissive preflight continues to use it for up to 24 hours — giving attackers a long exploitation window after a "fix" is deployed.
9:00am — Bug discovered, preflight cached (24h)
→
9:05am — CORS config patched on server
→
Attacker's browser still uses cached preflight until 9:00am tomorrow
Browser → OPTIONS /api/settings (preflight)
Server → 204 No Content
Access-Control-Max-Age: 86400 ← cached for 24h!
Browser → PUT /api/settings (real request, no preflight needed for 24h)

BUG #8 Unsafe Methods Allowed Cross-Origin

Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH on the DELETE endpoint means a trusted cross-origin page can trigger account deletion using the victim's session cookie. This is CSRF delivered via CORS misconfiguration.
Browser → OPTIONS /api/account
Server → Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH
Browser → DELETE /api/account with session cookie
Account deleted. Victim's data gone.

BUG #9 2026 NEW Private Network Access Header Misconfiguration

Chrome 98+ introduced Private Network Access (PNA). When a public website tries to reach a private network resource (localhost, 192.168.x.x), Chrome sends a preflight with Access-Control-Request-Private-Network: true. The server must explicitly opt in with Access-Control-Allow-Private-Network: true.

The bug: if the server sends that header to any origin that asks — not just trusted ones — it defeats PNA protection entirely.
🔒 What PNA is supposed to prevent:
Public website (evil.com) → tries to reach http://localhost:3000/api/internal
Chrome sends preflight with: Access-Control-Request-Private-Network: true
If server says: Access-Control-Allow-Private-Network: true without checking origin
→ Chrome allows the request → internal API exposed to any public website
