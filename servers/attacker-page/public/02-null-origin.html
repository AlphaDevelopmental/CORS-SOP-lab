Exercise 01 — Null Origin via Sandboxed iframe

Bug #3 — The origin you send can be "null" — and some servers trust it
Why this matters: An attacker embeds a sandboxed iframe anywhere on their page. The iframe's JavaScript sends Origin: null (browsers always do this for sandboxed iframes). If the server trusts null as an origin, the iframe can read the credentialed response — bypassing every hostname-based allowlist entirely.

BUG #3 Null Origin Trust

Browsers send Origin: null in these cases:
  • Sandboxed iframes — <iframe sandbox="allow-scripts">
  • Requests from file:// URLs
  • Cross-origin redirects (in some browsers)
  • Requests from data: URIs
The attack works in two steps. First we show you the raw null-origin request. Then we embed a sandboxed iframe that does the same thing — the way a real attacker would.

Sandboxed iframe (the real attack vector)

The iframe below has sandbox="allow-scripts". Its JavaScript will fetch /api/profile and post the result back to this page via postMessage. The browser automatically sends Origin: null from inside the sandbox.
⬇ Sandboxed iframe (invisible to victim, runs silently)
<iframe sandbox="allow-scripts" srcdoc="..."></iframe>

The iframe sends: Origin: null
Server responds: Access-Control-Allow-Origin: null + Allow-Credentials: true
Result: Profile data flows to the attacker's iframe → postMessage → this page
