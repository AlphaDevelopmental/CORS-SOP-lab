Exercise 02 — Regex & Subdomain Bypasses

Bugs #4, #5, #6 — when "almost correct" allowlists fail
Pattern: The developer tried to write an origin allowlist using string matching. Each attempt has a subtle flaw that lets a crafted origin slip through. These are some of the most common real-world CORS misconfigurations found in bug bounty programs.

BUG #4 Prefix Match Bypass

The server checks origin.startsWith('http://localhost:3001'). An attacker registers a domain that starts with the trusted string. The check passes. The server reflects it back with credentials allowed.
Vulnerable code (cors-config.js)
if (origin.startsWith('http://localhost:3001'))
Attack origins that pass
http://localhost:3001
http://localhost:3001.evil.com ✓
http://localhost:30010 ✓
We'll spoof the Origin header to: http://localhost:3001.evil.com

BUG #5 Suffix Match Bypass

The server checks origin.endsWith('bank.local') to allow any bank subdomain. But evilbank.local also ends with bank.local.
Vulnerable code (cors-config.js)
if (origin.endsWith('bank.local'))
Attack origins that pass
http://reports.bank.local
http://evilbank.local ✓
http://notabank.local ✓

BUG #6 Subdomain Wildcard Trust Pivot

The server trusts all *.bank.local subdomains, including localhost:3001 (which simulates reports.bank.local). If an attacker compromises any trusted subdomain (XSS, subdomain takeover), they can read the admin API using that subdomain's trusted-origin status.
